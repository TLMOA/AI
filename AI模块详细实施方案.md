# AI模块详细实施方案（面向JetLinks融合）

## 1. 文档目的与依据

本方案用于指导当前AI模块从“本地可运行验证”走向“可服务化交付并与JetLinks融合”的完整落地过程。

本方案依据以下材料整理：

- 智慧数据平台总体路线与阶段目标：智慧数据平台模块详细开发方案.pdf
- 功能清单与验收目标：功能描述.xlsx
- 当前工作区已有工程结构与历史实施进展（NiFi数据流、容器部署、输出路径、第五条标签流）

## 2. 项目目标与边界

### 2.1 总目标

构建可嵌入JetLinks生态的AI模块，形成“数据接入与转换 -> 标签与特征 -> 模型训练/推理 -> 预警反馈 -> 前端操作展示”的闭环能力。

### 2.2 当前团队边界

- JetLinks主平台由外部团队负责。
- 本团队（2人）负责AI模块建设与前端页面。
- 分工：
  - 成员A（你）：NiFi数据转换与流程编排。
  - 成员B：大模型/训练推理服务。

### 2.3 与JetLinks对接边界

- 本模块以独立后端微服务形态建设。
- 通过标准API、事件或回调与JetLinks进行系统融合。
- 不改动JetLinks核心代码，优先通过接口协议与配置实现接入。
- V1阶段前端先采用“脱离JetLinks联调的简化可改造版本”，通过适配层保留后续对接能力。

## 3. 需求拆解（对应功能描述.xlsx）

### 3.1 数据接入与管理

1. 支持数据库数据转为CSV/JSON/TSV。
2. 支持CSV/JSON/TSV互转。
3. 支持手动触发转换。
4. 支持自动定时转换：
	- 用户可见的自定义周期任务。
	- 企业静默任务（用户不可见）。
5. 支持标签功能：
	- 前端人工打标签。
	- 系统自动规则打标签。
6. 支持多格式副本输出。
7. 支持外部数据集手动导入。

### 3.2 算法与模型训练（这块只在前端列出操作窗口界面，后端由我的另一位团队成员负责）

1. 低代码/菜单化训练流程。
2. 至少5种常见算法。
3. 基于F1分数进行模型对比并选择最优模型。
4. 支持最优模型上线。
5. 支持私有模型上传与上线。

### 3.3 预测与预警

1. 使用最新业务数据进行预测。
2. 预测结果与阈值比较。
3. 异常触发预警（后续对接JetLinks工单/告警中心）。

### 3.4 前端能力

支持以上能力的页面化操作与展示，包括任务创建、进度查看、结果下载、模型管理与预警展示。

## 5. 目标架构（本期可落地版本）

### 5.1 逻辑分层

1. 数据流中枢层：NiFi（数据抽取、转换、标签、复制、调度）。
2. 智能服务层：模型训练/推理服务（FastAPI或同类）。

- 阶段V0.2：Ceph对象存储（通过存储抽象层切换）。

## 6. 分阶段实施方案

## 6.1 阶段A：V0.1 本地验证与服务化封装（优先）

### A1. 后端任务化接口

建议最小API集：

1. POST /api/jobs
	- 创建任务（转换/标签/训练/预测）。
2. GET /api/jobs/{jobId}
	- 查询状态（PENDING/RUNNING/SUCCEEDED/FAILED/CANCELED）。
3. GET /api/jobs/{jobId}/outputs
	- 查询输出文件列表。
4. GET /api/files/{fileId}/download
	- 下载文件。
5. POST /api/jobs/{jobId}/cancel
	- 取消任务。

### A2. 任务元数据表（建议）

最小字段：

- job_id、job_type、status、progress、error_message
- input_ref、output_refs、rule_version
- created_by、created_at、started_at、finished_at

### A3. NiFi流程治理

1. 统一命名规范（流程组、处理器、连接关系）。
2. 统一参数上下文（数据源、输出路径、增量列、规则版本）。
3. 标签规则版本化管理（rule_version）。
4. 失败路由和补偿重试（死信目录/重跑机制）。

### A4. 前端页面最小闭环

1. 任务创建页（手动触发）。
2. 任务列表与状态页（轮询/推送）。
3. 文件结果页（列表、预览、下载）。
4. 标签页（人工标签与自动标签入口）。

前端实现策略（V1）：

1. 不阻塞JetLinks联调，先实现可独立运行的前端壳。
2. 抽象API适配层（api adapter），业务页面只依赖前端内部接口。
3. 预留JetLinks适配器，后续仅替换适配层，不重写页面组件。
4. 页面配置化（菜单、字段、按钮权限）优先，保证可快速改造。

### A5. 与大模型同事的对接点

1. 训练数据格式契约（字段、类型、缺失值策略）。
2. 训练任务触发协议（同步/异步）。
3. 模型评估输出（至少包含F1、精确率、召回率）。
4. 模型版本与上线接口（activate/rollback）。

### A6. 验收标准（V0.1）

1. 数据转换：CSV/JSON/TSV互转全部可用。
2. 标签：人工标签、自动标签至少各1条完整流程。
3. 自动化：支持定时任务与手动任务并存。
4. 训练：至少5种算法比较并输出最佳模型。
5. 预警：预测异常可触发预警接口。
6. 前端：核心流程可全程可视化操作。

## 6.2 阶段B：V0.2 Ceph迁移与稳定化

### B1. 存储抽象层

实现统一接口：

- put/get/list/delete/signUrl

提供两套适配器：

- LocalStorageAdapter
- CephS3Adapter

### B2. 双写迁移策略

1. 过渡期本地+Ceph双写。
2. 定时校验文件哈希、条数、一致性。
3. 稳定后切换为Ceph主写，本地降级为缓存/回滚通道。

### B3. Ceph对象键规范

建议：

- bucket/jobs/{yyyy}/{mm}/{dd}/{job_id}/...
- bucket/models/{model_id}/{version}/...
- bucket/manifests/{job_id}.json

### B4. 验收标准（V0.2）

1. 业务逻辑不改，仅配置切换完成迁移。
2. 双写校验一致率达到100%。
3. 下载能力支持签名URL与权限控制。
4. 回滚开关可在分钟级完成切回。

## 6.3 阶段C：与JetLinks平台融合（联调交付）

### C1. 对接清单

1. 鉴权方式（Token/OAuth2/签名）
2. 统一数据对象（设备、测点、时序、告警）
3. 任务与告警接口契约
4. 错误码规范与重试策略
5. 日志追踪ID打通

### C2. 联调流程

1. 沙箱联调：接口可用性与字段验证。
2. 场景联调：故障预测闭环演示。
3. 压力联调：批量任务与并发下载。
4. 上线联调：灰度发布与回滚演练。

## 8. 风险与应对

1. 风险：NiFi状态与业务任务状态不一致。
	- 应对：后端统一状态机 + 定时对账任务。
2. 风险：规则变更导致标签口径不一致。
	- 应对：规则版本化，结果写入rule_version。
3. 风险：Ceph切换后历史下载断链。
	- 应对：统一file_id索引与storage_type映射。
4. 风险：2人团队资源紧张。
	- 应对：优先闭环路径，延后非关键优化项。
