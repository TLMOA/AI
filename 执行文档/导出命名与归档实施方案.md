# 导出命名与归档实施方案

## 一、目的
- 规范手动导出与定时导出的文件命名与存放位置，便于检索、审计与运维；
- 在不破坏现有下游消费（NiFi 流程）的前提下，逐步迁移到按任务归档的存储结构。

## 二、范围
- 手动导出（UI 上“导出并写入 NiFi 输出目录”）
- 定时导出（`export_jobs` 调度任务）

## 三、现状简述
- 手动导出当前写入：`nifi_data/output_{fmt}/`，文件名示例：`job_manual_410b505b_20260420_140225.csv`（可读性差）。
- 定时导出当前写入：`nifi_data/exports/{username}/{job_id}/...`，目录与文件命名不规范，难以检索与清理。

## 四、设计决策（已确认为需求）
1. 手动导出：保持写入 `nifi_data/output_{fmt}/`（兼容现有 NiFi 流程），但改用更有意义的文件名。
2. 定时导出：**仅写入** `nifi_data/exports/{username}/{job_id}/{YYYYMMDD_HHMMSS}/`（其中 `job_id` 为前端创建的 `export_jobs.id`，`username` 为登录用户名）；不写回 `output_{fmt}`。

## 五、命名规范
### 手动导出（实时输出）
- 目标格式（推荐）：
  - `export_manual_{username}_{table}_{YYYYMMDD_HHMMSS}.{ext}`
  - 示例：`export_manual_yhz_sensor_20260420_140225.csv`
- 可选（若需额外追溯信息）：在末尾追加短 id：`..._{sid}.{ext}`。

### 定时导出（归档到 exports）
- 目录：`nifi_data/exports/{username}/{job_id}/{YYYYMMDD_HHMMSS}/`
- 文件名（放在时间目录内，推荐）：
  - `export_{job_id}_{table}_{YYYYMMDD_HHMMSS}.{ext}`
  - 示例：`export_3_sensor_20260420_140225.csv`
  - 可选：`export_{job_id}_{username}_{table}_{YYYYMMDD_HHMMSS}.{ext}`（包含用户名，便于直接人工查看）。

## 六、元数据（每次定时导出必须生成）
- 在同目录写 `meta.json`，字段建议（必含）：
  - `job_id`, `job_name`, `username`, `table`, `file_format`, `file_name`, `file_size`, `rows`, `created_at`（ISO8601）、`trace_id`, `where`/`source_query`, `incremental_from`, `incremental_to`

## 七、安全与规范
- 所有用于路径或文件名的用户输入（`username`, `table`, `job_name`）必须 sanitize：只允许 `[A-Za-z0-9_-]`，其它字符转为 `_`。
- 创建目录时设置权限：`owner=app_user`，`mode=0750`。

## 八、实现计划（具体修改点）
> 采用「先改手动导出文件名、再改定时导出归档写入」的步进策略，保证最小切面影响。

### 步骤 1：手动导出 — 改文件名（最小变更）
- 修改文件：`v1-backend/app/main.py`
  - 函数：`export_mysql`（并可能同步 `export_generic`）
  - 行为：构建新的文件名 `export_manual_{username}_{table}_{ts}`，写入 `NIFI_OUTPUT_DIR`；调用 `_register_and_return_meta` 以在后端 `files` 注册。
  - 注意：不删除或替换现有 `_export_latest` 行为（保持兼容）。

### 步骤 2：定时导出 — 仅写入 exports
- 修改文件：`v1-backend/app/export_worker.py`
  - 函数：`run_export_job`
  - 行为：
    1. 产生 `timestamp = now_ts()`，构建 `archive_dir = NIFI_BASE_DIR / 'exports' / username / job_id / timestamp` 并 `mkdir(parents=True)`。
    2. 将导出结果写入该 `archive_dir` 下，文件名按规范 `export_{job_id}_{table}_{timestamp}.{ext}`。
    3. 在同目录写 `meta.json`（见上）并 `_append_audit`；更新或写 `.state.json`（保留增量状态）。
    4. **不写回 `NIFI_OUTPUT_DIR`**（按你要求，仅写入 exports）。

### 补充改动
- 在后端通用入口 `export_generic` 中，保证 `db_config` -> `run_export_job` 时传入 `owner_id/username` 与 `job_id`（若为手动触发，可构造 `job_id`）。
- 增加或复用 `ensure_export_dir` 与 `_save_state/_load_state` 来管理 state 与 meta 写入（现有 `export_worker.py` 已有部分支持）。

## 长期方案（认证优先，兼容回退）

为保证导出文件名中的 `username` 能准确反映当前登录用户（例如 `admin`），并兼顾定时任务与无登录场景，建议采用以下长期方案：

1. 优先来源规则：
  - 首选：后端从请求中的 JWT cookie（`access_token`）解析出用户名，使用 `_get_current_user_from_token` 获取 `sub`。
  - 其次：请求体中的 `owner_id`（由前端在必要时传入，作为兼容回退）。
  - 最后：若两者都不可用，则使用 `unknown` 或配置的默认值。

2. 后端改动（主要修改点）：
  - 在 `v1-backend/app/main.py` 的 `export_generic` 与 `export_mysql` 中，优先调用 `_get_current_user_from_token(request.cookies.get("access_token"))` 获取用户名并作为 `owner_id`。仅在该值为空时使用 `body.owner_id` 或默认值。
  - 生成文件名时统一使用通过上述优先级解析得到的 `owner_id`，并对其进行 sanitize（允许 `[A-Za-z0-9_-]`，其它字符替换为 `_`）。
  - 对定时任务（`run_export_job`）确保接收到 `owner_id`（若作业来自 UI 则由前端注入；若来自调度系统由调度元数据携带）。

3. 前端改动（兼容与增强）：
  - 在 UI 触发导出或触发定时任务前，优先调用 `/api/v1/auth/me` 获取当前用户名并把 `owner_id` 写入请求体（作为回退，同后端优先使用 cookie）。
  - 将现有硬编码 `runBy: "user-001"` 等替换为动态获取的用户名或移除此默认值。

4. 测试与验证：
  - 单元测试：覆盖 `export_generic` 在存在/不存在 cookie、存在/不存在 `owner_id` 时的文件名生成结果。
  - E2E：登录不同用户（如 `admin`、普通用户），触发手动导出和定时触发，验证 `nifi_data/output_{fmt}` 与 `nifi_data/exports/...` 中的文件名包含正确用户名。

5. 部署与回滚：
  - 分阶段部署：先部署后端改动（仅改名逻辑），观察日志与导出产物；确认后再部署前端改动。每步保留回退补丁。 
  - 打开运行时日志（导出操作记录 owner 来源与 traceId），便于监控与问题回溯。

6. 审计与文档：
  - 在 `meta.json` 与审计日志中显式写入 `owner` 字段与 `owner_source`（`cookie`/`body`/`fallback`），便于稽核。
  - 更新本文件与前端使用说明，记录字段优先级与兼容策略。

采用此长期方案后，手动导出文件名会以真实登录用户名命名（如 `export_manual_admin_sensor_20260420_140225.csv`），解决当前 `user-001` 默认值问题，并保持对无登录或自动调度场景的兼容性。

## 九、测试与验收标准
1. 手动导出：触发导出，检查 `NIFI_OUTPUT_DIR` 下文件名为 `export_manual_{username}_{table}_{ts}`，并可通过 UI 下载或后端 `files` 注册查询到。下游 NiFi 能正常消费。
2. 定时导出：为某 `job_id` 触发两次，检查 `nifi_data/exports/{username}/{job_id}/{ts}/` 两个时间目录存在，每目录包含数据文件 + `meta.json`；检查 `.state.json` 更新（若增量）。
3. 权限与清理：创建的目录权限正确；清理脚本能在删除前输出审计摘要。

## 十、回滚方案
- 每个补丁单独提交，变更点小且可回退：
  - 手动改名失败：回退 `export_mysql` 变更，旧文件名行为立即恢复。
  - 定时改造失败：回退 `export_worker.run_export_job` 变更，保留旧 `state.json` 与 `exports` 内容。

## 十一、执行时间估算与交付
- 步骤 1（手动改名）：约 1 小时（包含单元/手动验证）。
- 步骤 2（定时改造 + meta）：约 2-3 小时（包含测试）。
- 迁移脚本/清理脚本（可选）：1-2 小时。
