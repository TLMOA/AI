# 静默兜底导出实施方案（Silent Export）

版本：v1.0
作者：xy
日期：2026-04-28

**结论**
- 当管理员在内部管理页为某个工厂（tenant）启用时，系统将：
  1) 立即对该工厂数据库的每个表生成各自的 CSV 快照文件；
  2) 按设定周期（默认每天）检测每表增量并把新增行追加到对应 CSV 文件末尾；
  3) 全程遵守原子写入、审计记录、权限与保留策略。

**目标与范围**
- 目标：提供一条可控的“兜底”路径，在用户查询不可用或业务异常时保证表级数据可用性。 
- 范围：仅针对启用 `silent_export` 的 tenant；每个数据库表对应单独 CSV 文件（不是整库合并）。

**功能说明（要点）**
- 启用方式：内部管理页按 tenant 粒度的开关（默认关闭），仅 `system-admin`/ops 可配置。
- 首次导出：开启后立即对每个表做全量导出，生成 `<table>_silent_export.csv`。
- 日常追加：按计划任务（`silent_export.cron`）每天检测并追加新增行至对应文件（追加行为原子化）。
- 新增判定：优先使用 `incremental_marker_column`（如 `updated_at` 或自增主键）；若无则退化为行数/哈希差异策略并记录不确定性。
- Schema 变更：若列集合/顺序变化，创建新文件并在 `.meta.json` 中记录 schema diff，旧文件保持不变以避免破坏消费方。

**文件与目录规范**
- 基路径：`nifi_data/silent_exports/<tenant>/`
- 临时目录：`.../tmp/`（用于原子写入）
- 最终文件名：`<table>_silent_export.csv`（首次快照），若 schema 变更则：`<table>_silent_export_YYYYMMDD.csv`。
- 每个最终文件同时写入 `<file>.meta.json`，示例字段：
  - `traceId, jobId, operator, triggerReason, last_export_marker, rows_exported, timestamp, schema_hash`

**管理员界面与 API（建议）**
- 管理页位置：内部管理页“工厂配置”行内增加 `Silent Export` 状态与切换开关，旁边放“立即触发一次”按钮（需二次确认）。
- 后端 API：
  - `GET /api/internal/tenants/{tenant}/silent-export` —— 读取状态与配置（仅 admin）。
  - `POST /api/internal/tenants/{tenant}/silent-export` —— 更新配置（enable/disable/cron/retention）。
  - `POST /api/internal/tenants/{tenant}/silent-export/trigger` —— 管理员手动触发一次（演练）。
- 审计：所有配置变更与手动触发写入审计日志（who/when/from-ip/old->new）。

**调度与执行器设计**
- 调度：在 `v1-backend/scheduler.py` 中新增任务类型 `silent_export`，支持 cron 配置与最大并发控制（`silent_export.max_concurrent`）。
- 执行器职责（`v1-backend/executors/silent_export.py`）：
  1. 读取 tenant 配置与表清单；
  2. 对每个表检查 `last_export_marker` 并计算增量查询；
  3. 导出新增行到临时文件（`tmp/<table>.append.tmp`），fsync，合并或重命名到最终文件；
  4. 更新 `.meta.json` 与持久化 `last_export_marker`（存储可位于 DB 表或配置存储）。
- 并发与锁：按表粒度加执行锁（文件锁或 Redis 分布式锁），幂等键为 `jobId+tenant+table`。

**写入与原子性**
- 写入流程：
  1. 生成 `tmp/<table>.<timestamp>.csv` 并写入新增行；
  2. 调用 `fsync`、验证 checksum；
  3. 原子合并或重命名到 `.../<table>_silent_export.csv`（若为追加，可以先重命名 tmp->append.tmp，再原子拼接）；
  4. 写入/更新 `<file>.meta.json`。 

**幂等性与失败恢复**
- 执行器保存 `jobId` 与 `last_export_marker`，重试时根据 `jobId` 判断已完成的部分避免重复追加。
- 失败策略：重试（指数退避，最大 3 次），若仍失败写入 `silent_exports_failures/` 并触发告警，保留 tmp 文件供人工检查。

**保留与归档策略**
- 本地保留：`silent_export.retention_days`（默认 7 天）；过期数据自动归档到对象存储并可选加密；归档完成后删除本地副本。

**安全与权限**
- 默认关闭；只有拥有 `system-admin` 权限账号可启用/触发。 
- 写入账号为受限服务账号，目录 ACL 仅允许运维/受权服务读取。 
- 建议：对导出文件做静态加密或目标存储端加密；敏感列可支持列级脱敏（配置驱动）。

**监控与告警**
- 指标：`silent_export.count`, `silent_export.success.count`, `silent_export.failure.count`, `silent_export.rows_appended`, `silent_export.duration.p95`。
- 告警：连续 N 次失败或磁盘占用超阈值触发 P1/P0 告警；异常 schema 变更记录并告警。

**契约测试与验收**
- 必须加入 `contract-tests`：
  - 首次全表导出（文件格式/列/命名/元数据）；
  - 日增量追加（rows_appended 与 marker 更新）；
  - schema 变更场景（新文件生成）；
  - 并发触发与幂等验证；
  - 管理端启停与手动触发权限检查。

4. `contract-tests/silent_export_tests/`（契约测试用例）
5. 配置示例文件（`config/silent_export.example.yml`）

2. 开发样板（1-2 天）—— 新增执行器、管理 API 与配置；

**风险与缓解（摘要）**
- 数据泄露风险：默认关闭 + 加密 + 最小权限 + 审计。 
- 磁盘占用风险：严格 retention + 自动归档。 
- 误追加/重复风险：增量 marker + 幂等键 + 表级锁。
