# 联邦学习技术路线调查报告（V1）

- 日期：2026-04-15
- 适用项目：IoT AI 模块（当前 FastAPI + NiFi + 前端任务编排）
- 调研目标：评估“数据静默传输”替代路线，验证“数据不出本地”的联邦学习（Federated Learning, FL）可行性，并给出开源试点与后续算法合入路径。

## 1. 结论（Executive Summary）

✅ **建议立即启动联邦学习 PoC（首选 Flower）**，作为对当前“静默传输数据”路线的替代/补充能力。

✅ **核心收益明确**：原始数据留在用户侧，仅上传模型参数/梯度更新，可显著降低用户对“数据被拿走”的抵触。

⚠️ **风险需正视**：联邦学习不是“绝对隐私”，模型更新仍可能泄露信息；需引入安全聚合、差分隐私、传输加密与审计。

✅ **技术选型建议**：
- 第一阶段（2~4周）优先 Flower，原因是 Python 生态友好、上手快、社区活跃、对你们现有 FastAPI 架构改造成本最低。
- 第二阶段（4~8周）按场景升级：
  - 合规/多机构协作偏重：评估 FATE
  - 强安全与生产治理偏重：评估 NVFlare

## 2. 现状与问题定义

当前路线以中心化方式汇集数据（即便是“静默”），业务上可能触发以下问题：

1. 用户信任风险：用户感知为“数据被后台拿走”。
2. 合规压力：敏感数据跨边界流动，审计与合规成本高。
3. 商业阻力：潜在合作方（尤其企业）常要求“数据不出域”。

联邦学习的价值在于把训练计算推到数据本地，中心侧只做参数聚合和模型编排。

## 3. 开源方案调查结果

## 3.1 Flower（推荐首选）

- 开源协议：Apache-2.0
- 特点：
  - 框架无关（PyTorch/TensorFlow/sklearn/XGBoost 等）
  - 快速上手（`pip install flwr` + Quickstart）
  - 支持模拟与真实部署，样例和 baseline 丰富
  - 支持跨端（含移动端 SDK 文档方向）
- 适配你们项目的优势：
  - 与现有 Python/FastAPI 体系贴合，PoC 周期短
  - 可先从“横向联邦 + FedAvg”落地，再增量增强安全
- 风险：
  - 工程化治理能力需自行补齐（身份、密钥、审计、策略中心）

**结论**：最适合“先跑通，再规模化”的第一阶段。

## 3.2 NVFlare（生产治理能力强）

- 开源协议：Apache-2.0
- 特点：
  - 明确面向生产部署，含部署/安全/运维体系
  - 支持 DP、同态加密、PSI、审计、策略控制
  - 支持云/本地/K8s、多种工作流与行业用例
- 适配建议：
  - 当你们进入多组织、多环境、强安全审计阶段可重点引入
- 风险：
  - 学习与集成成本高于 Flower

**结论**：适合作为中后期企业级方案候选。

## 3.3 FATE（工业级联邦与隐私计算）

- 开源协议：Apache-2.0
- 特点：
  - 工业级联邦框架，强调隐私计算（MPC/同态等）
  - 生态较完整（FATE-Flow、Serving、Board、KubeFATE）
  - 支持横向/纵向联邦、跨机构协作
- 适配建议：
  - 若后续出现“多机构联合建模 + 强监管合规”需求，优先评估
- 风险：
  - 架构与部署复杂度较高，不适合最小成本试错阶段

**结论**：适合“合规驱动、跨机构”场景，不建议作为第一步。

## 3.4 TFF / FedML / PySyft（补充观察）

- TFF：研究属性强，适合算法实验和新算法验证，工程生产化通常需额外平台层。
- FedML：覆盖面广，但近年公开仓库活跃度与产品重心变化较大，需谨慎评估长期路线稳定性。
- PySyft：偏“远程数据科学/隐私计算”治理范式，不是你们当前最短路径 FL 编排方案。

## 4. 选型建议（面向当前团队与架构）

### 4.1 推荐路线

✅ **路线A（立即执行）**：Flower PoC（横向联邦 + FedAvg）

✅ **路线B（中期增强）**：在 Flower 上补安全与治理能力（安全聚合、DP、审计、准入）

✅ **路线C（按业务触发）**：若进入多机构高合规，评估 NVFlare/FATE 的替换或并行。

### 4.2 不建议路线

⚠️ 直接上复杂工业平台（FATE/NVFlare）再做业务验证：
- 成本高、周期长，容易阻塞你们当前 V1/V2 节奏。

## 5. 与现有技术栈的融合设计（不传原始数据）

## 5.1 目标架构（建议）

1. 中心编排端（你们现有后端扩展）
- 职责：联邦任务创建、轮次调度、客户端选择、全局模型版本管理、审计日志。

2. 客户端训练代理（部署在用户侧）
- 职责：读取本地数据训练本地模型，只上报模型更新（权重差分/梯度/指标）。

3. 安全聚合层
- 职责：聚合多端更新，避免中心侧直接看到单点更新（逐步接入 secure aggregation）。

4. 模型仓库与发布层
- 职责：保存全局模型版本，供预测服务拉取发布。

## 5.2 与现有模块的关系

- NiFi：继续负责本地数据预处理/特征构造，但输出留在本地训练代理可访问域。
- FastAPI：新增联邦任务 API（`/api/fl/jobs`、`/api/fl/rounds`、`/api/fl/models`）。
- 前端：新增“联邦训练任务中心”页面，显示参与节点、轮次、聚合指标、模型版本。
- 算法端：将现有训练脚本改造为“可联邦化 client_train() + server_aggregate()”接口。

## 6. 后期算法端合入方案（重点）

## 6.1 改造原则

1. 训练逻辑与联邦编排解耦。
2. 模型定义尽量保持不变，新增联邦适配层。
3. 指标定义统一（F1、Precision、Recall、AUC），支持中心端跨轮次对比。

## 6.2 建议接口

- `client_train(local_data_ref, global_model, train_config) -> model_update, local_metrics`
- `server_aggregate(updates, strategy_config) -> new_global_model`
- `evaluate(global_model, validation_ref) -> metrics`

## 6.3 算法落地优先级

1. 先改一类最稳模型（如逻辑回归 / XGBoost / 轻量神经网络）做闭环。
2. 再扩展到你们计划中的5类算法，并统一评估面板。
3. 最后接入“最优模型自动上线/回滚”机制。

## 7. 安全与合规注意事项

⚠️ 联邦学习仅保证“数据不出域”，不自动等于“零泄露”。建议最小安全基线：

1. 传输安全：TLS 双向认证。
2. 身份与准入：节点证书 + 白名单 + 任务签名。
3. 聚合安全：优先引入安全聚合（Secure Aggregation）。
4. 隐私增强：差分隐私（DP）参数纳入可配置策略。
5. 审计追踪：记录每轮参与者、模型版本、聚合摘要、异常事件。
6. 最小上传原则：仅上传必要更新与指标，禁止原始样本/可逆特征外传。

## 8. PoC 实施计划（建议 4 周）

### 第1周：最小闭环
- 完成 Flower 服务端 + 2~3 个本地客户端样例
- 跑通 5~10 轮联邦训练
- 输出全局模型与轮次指标

### 第2周：接入现有后端
- FastAPI 新增联邦任务 API
- 与任务状态机打通（PENDING/RUNNING/SUCCEEDED/FAILED）
- 前端展示联邦任务基础状态

### 第3周：接入算法端
- 至少 1 种现有业务算法联邦化
- 对齐 F1 为主的评估标准
- 支持模型版本登记与下载

### 第4周：安全与演示
- TLS、节点准入、基础审计
- 演示“数据不出域”训练流程（可对外沟通）
- 输出阶段评估报告与下一阶段预算

## 9. 成本与收益评估（定性）

✅ 收益：
1. 用户信任提升（数据不离端）。
2. 合规压力下降（减少原始数据跨域）。
3. 商业可谈判性增强（面向政企更友好）。

⚠️ 成本：
1. 通信与轮次调优复杂度上升。
2. 非IID数据导致收敛与公平性挑战。
3. MLOps 复杂度增加（多端版本管理、断线重试、安全审计）。

## 10. 决策建议

✅ 建议立即立项“联邦学习 PoC（Flower）”，并把它定义为“用户信任友好能力”而非一次性替代。

✅ 立项里程碑建议：
1. M1：2周内跑通 PoC（3节点、10轮、可视化指标）。
2. M2：4周内接入现有任务系统和至少1种业务算法。
3. M3：8周内完成安全增强并评估是否切换/并行 NVFlare 或 FATE。

⚠️ 若 PoC 指标不达标（收敛慢、效果差），建议采用“混合策略”：
- 对高敏数据场景使用联邦学习；
- 对低敏且需高时效场景保留中心化训练。
