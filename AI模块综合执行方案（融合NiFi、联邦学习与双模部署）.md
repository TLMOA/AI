## 一、总体架构设计

### 1.1 系统架构概览
```
前端层（UI/JetLinks适配层）
    ↓
统一编排层（FastAPI后端）
    ↓
执行器层（Local执行器 / NiFi执行器）
    ↓
数据流层（统一逻辑目录输出）
    ↓
Client Agent（Sidecar）读取本地产物
    ↓
训练层（中心训练 / 联邦学习）
```

### 1.2 部署模式架构
#### 公有化部署（数据存公司数据库）
- 工厂数据 → Client Agent → 直接上传 → 公司数据库 → 中心训练

#### 私有化部署（数据存工厂数据库）
- **方式A**：工厂数据 → Client Agent → 安全拉取 → 公司数据库 → 中心训练
- **方式B**：工厂数据 → Client Agent → 本地联邦训练 → Flower Server → 仅上传模型参数

### 1.3 数据处理双轨制
- **Local执行器**：轻量级本地数据处理，已实现基础功能
- **NiFi执行器**：可编排的数据流处理，**核心功能未实现**
- 前端通过配置按钮透明切换执行模式，输出至统一逻辑目录

## 五、关键风险与应对策略

### 5.1 技术风险
#### NiFi集成风险
- **风险**：NiFi执行器功能未经验证，可能存在兼容性问题
- **应对**：优先完成实验室验证，建立回滚机制

#### 联邦学习收敛风险
- **风险**：Non-IID数据可能导致训练效果不佳
- **应对**：实施混合策略，保留中心训练作为备选

### 5.2 资源风险
#### 团队资源紧张
- **风险**：2人团队同时推进多个复杂模块
- **应对**：聚焦核心路径，分阶段交付，延后非关键功能

### 5.3 集成风险
#### 平台融合复杂度
- **风险**：JetLinks集成可能遇到接口兼容性问题
- **应对**：早期介入接口设计，建立契约测试
