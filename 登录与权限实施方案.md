# 登录与权限实施方案

## 概要
- 目标：为网站新增用户注册、登录与管理员权限控制。**登录为全站统一入口**，一个页面为登录页面，用户登录后可在站点任意页面获得登录态；管理员登录时在全站导航中显示“内部管理”跳转链接，普通用户不可见且无法访问内部管理页。仅预置管理员 `admin`（初始密码 `123456`），管理员不可通过注册创建。
- 原则：不影响其他现有功能 —— 所有改动应最小化侵入，使用独立表名与明确接口，部署前务必备份数据库并低峰期发布。

## 总体方案
- 存储：复用现有 NiFi 数据库（使用项目现有 SQLAlchemy 引擎），新增独立用户表 `iot_users`（避免与 NiFi 表冲突）。
- 认证方式：后端使用用户名/密码登录，密码采用 `bcrypt` 哈希，登录后签发短期 JWT 并通过 `HttpOnly Secure` Cookie 传递（推荐）。
 - 认证方式：后端使用用户名/密码登录，密码采用 `bcrypt` 哈希（fallback 为 PBKDF2 在依赖不可用时使用），登录后签发短期 JWT 并通过 `HttpOnly Secure` Cookie 传递（推荐）。API 挂载在 `prefix=/api/v1` 下，最终接口为 `/api/v1/auth/*`。
 - 前端导航规则：全站导航组件在页面渲染或 SPA 路由变更时调用 `GET /api/v1/auth/me`，依据返回的 `is_admin` 字段决定是否显示“内部管理”链接；后端仍以权限检查为准，前端仅负责显示隐藏。
- 权限：`iot_users` 含字段 `is_admin`，后端中间件在请求上下文注入当前用户信息并用于权限校验。所有内部管理页面/API 增加 `require_admin` 校验。

## 不影响现有功能的设计原则（必须遵守）
1. 表名独立：使用 `iot_users`（或 `app_users`）表，绝不修改 NiFi 原有表结构或数据。 
2. 连接复用、逻辑隔离：复用 `v1-backend/app/engine_factory.py` 提供的数据库连接，但将认证逻辑与现有导出/调度逻辑严格分离、仅通过 API 调用交互。 
3. 启动兼容：应用启动时仅执行 `CREATE TABLE IF NOT EXISTS` 或轻量迁移，不会对现有表做DROP/ALTER等破坏性操作。 
4. 备份强制：上线前需备份数据库快照并在低峰期切换。

## 数据库设计（最小字段）
- 表名：`iot_users`
- 字段：
  - `id` INTEGER PRIMARY KEY AUTOINCREMENT
  - `username` TEXT UNIQUE NOT NULL
  - `password_hash` TEXT NOT NULL
  - `is_admin` BOOLEAN NOT NULL DEFAULT 0
  - `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP

## 最小 API 设计
- `POST /api/v1/auth/register` — 注册普通用户（body: username,password）
- `POST /api/v1/auth/login` — 登录（body: username,password），成功后返回 200 并设置 HttpOnly Cookie
- `POST /api/v1/auth/logout` — 登出（清除 Cookie）
- `GET /api/v1/auth/me` — 返回当前用户信息（含 `is_admin`）

## 管理端权限保护
- 在后端内部路由统一添加 `require_admin` 装饰器/中间件：若未登录或 `is_admin==False` 返回 403。前端 `internal.html` 在加载时请求 `GET /api/auth/me`，若 `is_admin` 不为 true，则重定向到登录页或显示无权限提示。

## 启动与预置管理员
- 启动检查：应用启动时检查 `iot_users` 表是否存在；若不存在则创建。检查是否存在用户名 `admin`，若无则插入 `admin`（密码为哈希后的 `admin`）。
- 安全提示：预置管理员密码仅为启动方便，部署后应尽快修改。文档中明确说明首次上线后的改密步骤。

## 安全要点
- 密码哈希：使用 `bcrypt` 或 `argon2`（建议 `bcrypt` + `passlib`）。
- 传输：强制 TLS/HTTPS。 
- Cookie：`HttpOnly`, `Secure`, `SameSite=Lax`（或更严格）。
- JWT：短期过期（例如 15 分钟），如需要长期会话使用刷新 token 或短期 cookie + 服务端会话存储。

## 兼容与风险控制
- 风险：在同库建表可能导致锁或权限问题，部署前请备份并在低峰期执行。 
- 兼容措施：建表语句使用 `IF NOT EXISTS` 并在事务内执行；不对现有表执行 ALTER/DELETE。 

## 测试计划
1. 单元测试：用户模型、密码哈希、登录接口、注册接口、admin 检查装饰器。 
2. 集成测试：使用 SQLite 临时 DB 或开发 NiFi DB 副本，验证注册、登录、权限隔离。 
3. E2E：前端登录/注册流程、`internal.html` 权限拒绝/允许场景测试。 

## 部署步骤（最小影响发布流程）
1. 备份当前 NiFi 数据库快照。 
2. 在测试环境部署并验证（包含 DB 副本）。 
3. 在低峰期部署至生产：停服务（或使用滚动升级）、执行建表脚本、启动并观察日志。 
4. 验证：使用 `admin/admin` 登录并访问内部页；用普通用户验证无法访问内部页。 
5. 如出现问题：按回滚策略恢复数据库与旧服务。 

## 回滚策略
- 若部署出现不可接受故障：
  1. 立即停止新版本服务。 
  2. 恢复数据库备份（若建表无破坏性则可跳过）。 
  3. 启动回滚前的服务版本。 

## 验收标准
- 普通用户能注册并登录，但在全站导航中**看不到**“内部管理”链接，直接访问 `internal.html` 将被后端拒绝或重定向。 
- 管理员使用预置管理员（用户名 `admin`，初始密码 `123456`）能登录并在全站导航看到“内部管理”跳转，同时能访问内部管理页。 
- 密码以哈希形式存储于 `iot_users` 表。 

## 估时与负责人建议
- 估时（MVP）：2-3 工作日（后端实现 + 简单前端集成 + 测试）。
- 建议负责人：后端开发一名（改动现有后端）、前端一名（登录/注册页面与 `internal.html` 权限校验）。
