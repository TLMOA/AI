
# 任务 — 精简版（仅保留两项需求）

## 概要
- 聚焦两个功能：
	1) 用户侧“定时自动转换”——仅支持 CSV / JSON / TSV；前端配置频率与保存位置，后端执行并生成文件。 
	2) 管理端“静默传输控制”——管理员可启停/修改/触发静默传输任务，普通客户不可见传输细节。

## 已启动实施（2026-04-13）
- ✅ 单工厂基线已落地：后端 `export_jobs` 增加 `factory_id` 字段，默认值 `factory-001`。
- ✅ API 已兼容：创建/查询/更新导出任务已支持 `factory_id`，查询支持按工厂筛选。
- ✅ 调度链路已打通：任务创建、启动恢复、手动/自动执行时均透传 `factory_id`。
- ✅ 内部页接收最小接口已落地：
	- `POST /api/v1/internal/factory-reports`（工厂侧上报批次结果）
	- `GET /api/v1/internal/factory-reports`（内部页按工厂查询，支持 limit）
- ✅ 导出审计增强：`audit.log` 记录加入 `factory_id` 字段，便于未来多工厂汇总。

### 当前约定（单工厂）
- 当前项目固定视作一个工厂，统一使用 `factory_id=factory-001`。
- 后续扩展多工厂时，只需在工厂部署侧替换 `factory_id`，中心端接口与表结构保持兼容。

### 内部页访问方式（已可用）
- 访问地址：`http://<你的前端IP>:5174/internal.html`（本机可直接用 `http://127.0.0.1:5174/internal.html`）。
- 数据来源：`GET /api/v1/internal/factory-reports`。
- 当前默认筛选工厂：`factory-001`。

### 内部页可操作能力（已可用）
- `GET /api/v1/internal/factory-jobs`：按工厂查询可操作的导出任务。
- `POST /api/v1/internal/factory-jobs/{job_id}/fetch`：内部页一键触发“获取数据”。
- 触发后会自动追加内部上报记录，可在上报表格实时查看执行结果。

## 假设与前提
- 现有系统已支持手动转换为 CSV/JSON/TSV。
- 初期不新增其它文件格式。
- 静默传输需使用 TLS 与服务到服务认证（JWT 或 mTLS）。

## 需求一：前端可配置的定时自动转换（CSV/JSON/TSV）

目的：允许用户在前端为现有手动转换创建定时任务（配置频率、格式和保存位置），并支持手动触发。

要点实现（MVP）
- 前端：新增“定时任务”面板，字段：频率（下拉/cron）、格式（CSV/JSON/TSV）、保存位置、本地/S3/SFTP、启用开关、手动触发。
- 后端：新增 `export_jobs` 表并提供 CRUD + trigger API。
- 调度：APScheduler（MVP），后续可用 Celery+beat 替换。
- Worker：实现 MySQL→CSV/JSON/TSV 导出并存储到目标位置。

API 示例
- POST /api/export-jobs
- PUT /api/export-jobs/:id
- GET /api/export-jobs
- POST /api/export-jobs/:id/trigger

安全与验收
- 权限限制：仅允许创建/管理自己数据的用户或管理员。
- 验收：前端可创建任务，调度器生成文件，手动触发即时生效。

估时（MVP）：2-3 天。

## 需求二：管理端静默传输控制页面（Admin UI）

目的：提供管理员界面管理静默传输任务（启停、修改频率、手动触发、查看审计），普通客户不可见传输细节。

要点实现（MVP）
- UI：任务列表、编辑/启停、手动触发、审计查看（仅管理员）。
- 后端 API（admin only）：GET/PUT/POST trigger/audit 对应 `/api/silent-jobs`。
- 静默传输：Worker 将文件通过 HTTPS POST（或私有 S3 + 服务到服务预签名）发送到我方接收端；前端与普通日志不展示传输细节。

安全要点
- 访问控制：仅 `admin` 或 `silent:manage` 权限可访问。
- 传输：TLS + JWT/mTLS；凭证加密或使用 Vault。
- 审计：记录摘要信息（仅管理员可见），普通日志不包含文件名/内容。

验收：管理员可管理静默任务并验证文件已安全到达接收端；客户界面不显示静默详情。

估时（MVP）：1.5-2 天。

## 存储目录与文件管理
- 默认保存位置：项目内统一使用 `nifi_data` 目录作为导出文件根目录。
	- 建议路径：`nifi_data/exports/{customer_id}/{job_id}/YYYYMMDD_HHMMSS/`。
	- 启动时或创建任务时自动确保目录存在并设置合适权限（owner=app_user，mode=750）。
	- 文件命名建议包含 `job_id`、时间戳与格式后缀，例如 `job_123_20260410_150000.csv`。
	- 轮转与清理：支持基于天数或总占用空间的自动清理策略（config 可配置），并在清理前将将要删除的文件摘要写入安全审计日志以便追踪。
	- 对象存储选择：若配置为 S3/S3-compatible，则上传后可删除本地副本或保留一份，取决于 `retain_local_copy` 配置。

## 支持的数据库（超出 MySQL 的实现细节）
- 目标：MVP 支持 MySQL，同时预留并实现对市面常用数据库的接入层，便于后续扩展为多数据库导出。
- 推荐实现方式：使用 `SQLAlchemy` 作为统一抽象层，配合不同驱动（DBAPI）以支持多种数据库。具体建议：
	- MySQL / MariaDB: 驱动 `pymysql` 或 `mysqlclient`。
	- PostgreSQL: 驱动 `psycopg2` 或 `psycopg[binary]`。
	- Microsoft SQL Server: 驱动 `pyodbc`（或 `pymssql` 作为备选）。
	- Oracle: 驱动 `cx_Oracle`（需要 DB 客户端依赖）。
	- SQLite（测试/小型场景）：使用 Python 标准库 `sqlite3`。
- 注意事项：
	- 不同数据库在分页/大表读取、时区、NULL 处理、编码、日期格式上存在差异，导出代码需统一规范化（例如将时间统一序列化为 ISO8601）。
	- 对于大表导出，优先使用分批分页（limit/offset 或基于主键范围）或游标流式读取以降低内存占用；Postgres 可考虑 server-side cursor；MySQL 可使用 iterator fetchmany。
	- 增量导出：建议通过 `updated_at` 字段或变更表（CDC）实现，不同 DB 的CDC方案差异较大，先以 `updated_at` 为主。
	- 连接配置：在 `export_jobs` 中保存 DB 连接字符串与加密凭证（仅管理员可查看），建议支持 DSN 格式并在运行时构造 SQLAlchemy 引擎。
